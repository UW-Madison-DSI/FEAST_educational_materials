Week 5 · Process • PR walkthrough
Honest question

Where did you shortcut?

Every team shortcuts under pressure. The question is: did it matter?

Coded before writing a spec?
Did you end up building the wrong thing? Or was the fix genuinely obvious enough that a spec would have been overhead?
Skipped adversarial review?
Did a bug slip through? Or was the change small enough that three layers was overkill?
Merged without all three layers?
Was it a time pressure decision? Would you do it again?
Accepted agent code without fully reading it?
Did it work? Did something subtle break later? How did you catch it?

This is how you calibrate. Some steps earn their cost every time. Some only matter above a certain complexity. Knowing the difference is engineering judgment, not a checklist.

Week 5 · Process • Shortcuts
Reflection

Three questions.

What took longer than expected?

Not what was hard. What was unexpectedly time-consuming. Understanding where your estimates were wrong is how you get better at estimating.

What was easier than expected?

When did the agent planning tool nail it? When did a task you dreaded take 30 minutes? What made the difference?

What would you redo?

Not at the process level. At the specific-task level. Different approach, different order, different scope.

Go around the room. One answer per question per person. Short and specific. "The CRS refactor took three times longer than I expected because..." is more useful than "it was hard."

Week 5 · Pipeline • Retrospective
Part four
The complete
workflow.
Assembling everything into one loop
The core loop

Spec. Plan. Execute. Verify.

Every piece of this, you have practiced. This is the assembled version.

1

Spec

You write 2-3 sentences. What, why, how to verify. The agent does not decide what to build.

2

Plan

Agent creates a phased plan from your spec. Each phase leaves the code in a working state.

3

Execute

One phase at a time. Do not batch. Implement, then stop.

4

Verify

Run tests. Review the diff. Check for over-engineering. Update the plan.

5

Document

ADR if you chose between approaches. Not every phase needs one.

The loop is adaptive. You update the plan after each phase. A plan that never changes means you are not paying attention. Verification is after every phase, not just at the end. See Guide 13 for the full reference.

Week 5 · Workflow • Core loop
Workflow selection

Not every task needs the full loop.

Task scope Workflow
Quick fix (< 30 min) Spec (1 sentence), implement, verify. No phasing needed.
Single feature (1-4 hours) Full spec-plan-execute-verify loop with phases.
Cross-cutting (multi-issue) Same loop, add multi-issue planning to sequence work.
Exploratory (unknown scope) Investigation phase first (read code, form hypothesis), then spec.

Choosing the right workflow is itself an engineering skill. When in doubt, default to phased execution. The overhead is small. The safety margin is large.

Beyond this project: This loop works with any agentic tool, not just the one you used here. The principles (spec first, phased execution, verify after each phase, explain every line) are tool-agnostic.
Week 5 · Workflow • Task matching
Making it automatic

Encode the workflow in your context file.

A workflow section in your CLAUDE.md tells the agent to follow the loop automatically.

Example: workflow section in CLAUDE.md
## Workflow
- Create a phased plan before writing code
- Each phase must leave the codebase in a
  working, testable state
- Run the full test suite after each phase
- Flag any phase that adds more than 200 lines
- Do not proceed to the next phase until the
  current one passes all tests

What this does

  • Programs the agent's behavior in natural language
  • Runs inside the LLM's context window
  • Flexible and easy to author
  • Interpreted, not executed mechanically

What this means

The context file is the spec. The agent is the executor. You are the verifier. Phased verification matters because you are the reliability layer.
