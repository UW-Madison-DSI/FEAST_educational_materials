Week 4 · Pipeline check-in • Health check
What to watch for

Patterns that slow teams down.

  • 1
    Reviews stack up
    PRs sit for days with no reviewer. Fix: Review others before asking for reviews. It is a team resource, not a favor.
  • 2
    Specs that never become code
    Well-written specs, but no implementation. Fix: The spec is not the deliverable. The merged PR is. Time-box spec writing.
  • 3
    Skipping the spec
    Jumping from issue to code with no plan. Fix: For anything beyond a trivial change, even two sentences in the issue comment is better than nothing.
  • 4
    Adversarial review debt
    The rotation falls behind. Fix: If you owe a review, it is your top priority today.

Part two
Manual vs.
auto-generated.
When to write, when to generate
Live demo

Generating an ADR from a diff.

Feed a PR diff to the agent. Ask it to write an ADR. Watch what it gets right and what it misses.

Prompt
Here is the diff for issue #30 (multi-store food access scoring).
Write an ADR documenting this decision. Use the template:
Title, Status, Context, Decision, Alternatives Considered,
Consequences.

What it gets right

  • Facts from the diff: what changed, where
  • Before/after behavior
  • File and function references
  • Technically accurate description

What it misses

  • Why you rejected alternatives
  • Unstated constraints the team discussed
  • Stakeholder concerns that shaped the choice
  • The reasoning behind "why not X?"
Takeaway: Auto-generated ADRs are good starting drafts. Feed the diff, the PR description, and the issue context. Then edit in the reasoning yourself.
Week 4 · Artifacts • ADR generation
Artifacts

When to write, when to generate.

Auto-generated

  • Captures what happened
  • Accurate about mechanics
  • Fast to produce
  • Good starting draft

Best for: initial ADR draft, changelogs, code documentation

Hand-written

  • Captures why it matters
  • Records rejected alternatives
  • Preserves team context
  • Explains "why not?"

Essential for: design tradeoffs, stakeholder constraints, priority decisions

Same principle as code generation:

The tool produces a first pass. You add the judgment. The "explain every line" rule applies to prose too.

Changelog from git log

hand-maintained

ROADMAP.md

  • What matters and in what order
  • Dependencies between milestones

A roadmap stays useful because someone updates it after each milestone, not because it was written once. It tracks what the team decided matters. Git log tracks what happened.

Week 4 · Planning at scale • Multi-issue
Approaches

Pick the tool that fits the question.

Approach Scope Input Good for
Single-feature planning One feature Issue + codebase Mapping code changes for one task
Multi-issue planning Project phase Set of issues + codebase Sequencing work, finding dependencies
Branch-level review Branch or PR set Diff against base branch Catching cross-cutting issues before merge
Hand-written spec / ADR Any Your reasoning Capturing why and what, not just how

No planning tool replaces human judgment about which goals matter. The tools map how. You decide what and why.

Week 4 · Design patterns • Five patterns
2-minute activity

Update your context file.

Open your project context file right now. Quick check:

Is it current?

Does it reflect what you have learned about the codebase since Week 1? If it still says "there are no tests" and you wrote tests in Week 3, update it.

Is it useful?

Start a fresh agent session. Does the context file give it enough to be helpful? What is it missing?

Is it honest?

Does it describe the codebase as it is, including rough edges? Or does it describe an idealized version?

The agent is only as good as the context you give it. A stale context file is worse than no context file, because it gives the agent false confidence about the state of the code.
