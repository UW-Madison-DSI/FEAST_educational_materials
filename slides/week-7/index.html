Judging the milestones

The same lens on every milestone.

We are sizing up the milestones and issues you drafted, not reviewing code.

Is the milestone real?

  • Demonstrable: you can show it
  • Testable: you know when it is done
  • Incremental: useful even if the track is not finished
  • Sprint-sized: one or two sprints, not a quarter

Does it trace up?

Every issue should connect:

issue → milestone → track → north star

If it does not trace, it is missing a milestone above it, or it does not belong in the roadmap yet.

"Make the UI better" is a wish. "Adding a store shows a progress bar, then a one-line summary of what changed" is a milestone. We reshape wishes into deliverables right here.

Working time

Break your milestones into issues.

  • 1
    Split into track groups
    Each track takes its own milestones. The milestone lead keeps the group moving.
  • 2
    For each milestone, list the issues it needs
    What has to be true for the milestone to be demonstrable? Each piece is one issue.
  • 3
    Write each issue so it stands alone
    Title, repo (backend or frontend), rough size (XS to XL), and a one-line acceptance check.
  • 4
    File them on the board
    Use the issue template, then set Repository, Milestone, and an owner on the org project.
~30 minutes in track groups, then we regroup. The bar for leaving today: every milestone we keep has at least its first issue filed, sized, and assigned.
